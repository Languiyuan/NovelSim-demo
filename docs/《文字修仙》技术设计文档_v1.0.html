3.5 Prompt Engineering 策略

分层 Prompt 架构

层级名称内容token 预算
L1System PromptAI 身份 + 全局规则 + JSON Schema~2000
L2Context Prompt世界大纲摘要 + 故事脉络摘要 + 结构化状态~1500
L3Task Prompt当前任务指令 + 约束条件 + 输出格式~800
L4History最近 5 节点摘要 + 玩家上次选择~500

总计单次调用 token 消耗约 4800 input + 1500 output，单次成本约 0.003 元（DeepSeek 定价）。

摘要压缩策略

  • 触发时机：每 5 个节点生成一次摘要，由 Agent 独立调用完成
  • 摘要格式：200-300 字，包含关键转折、NPC 关系变化、获得的重要物品
  • 保留原文：最近 2 个节点的原始文本始终保留在上下文中
  • 版本控制：摘要带版本号，新摘要生成后替换旧摘要

3.6 LangChain 集成方案

// agent/providers/langchain.provider.ts
import { ChatOpenAI } from '@langchain/openai';
import { SystemMessage, HumanMessage } from '@langchain/core/messages';
import { StructuredOutputParser } from '@langchain/core/output_parsers';

export class LangChainProvider {
  private llm: ChatOpenAI;

  constructor(config: AIConfig) {
    this.llm = new ChatOpenAI({
      model: config.model,               // deepseek-chat
      apiKey: config.apiKey,
      configuration: { baseURL: config.baseURL },
      temperature: 0.85,
      maxTokens: 2000,
      modelKwargs: { response_format: { type: 'json_object' } },
    });
  }

  async invoke(systemPrompt: string, userPrompt: string): Promise<string> {
    const result = await this.llm.invoke([
      new SystemMessage(systemPrompt),
      new HumanMessage(userPrompt),
    ]);
    return result.content as string;
  }

  // 带重试的 JSON 解析调用
  async invokeWithRetry<T>(
    systemPrompt: string,
    userPrompt: string,
    parser: StructuredOutputParser<T>,
    maxRetries = 2,
  ): Promise<T> {
    for (let i = 0; i <= maxRetries; i++) {
      try {
        const raw = await this.invoke(systemPrompt, userPrompt);
        return parser.parse(raw);
      } catch (e) {
        if (i === maxRetries) throw e;
        // 重试时追加"严格JSON"提示
        userPrompt += '\n\n[系统提示：上次输出格式错误，请严格返回合法JSON]';
      }
    }
    throw new Error('Unreachable');
  }
}

4.2 鉴权模块

流程

  1. 前端调用 wx.login() / my.getAuthCode() 获取临时凭证
  2. 后端用凭证换取 openid（微信）/ user_id（支付宝）
  3. 根据 openid 查找或创建用户记录
  4. 签发 JWT（access_token 2h + refresh_token 7d）
  5. 后续请求通过 AuthGuard 校验 JWT

安全设计

  • JWT 密钥存环境变量，生产环境使用 RS256
  • Refresh Token 单次使用，存 Redis 做轮转
  • Token 黑名单机制（用户主动退出时作废旧 Token）

function rollCharacter(): CharacterRoll {
  const roll = (min, max) => Math.floor(Math.random() * (max - min + 1)) + min;
  const jing = roll(30, 80);
  const qi = roll(30, 80);
  const shen = roll(30, 80);

  return { name: generateName(), jing, qi, shen, talent, wisdom, luck, maxAge };
}

4.3.3 战斗服务 (BattleService)

// 战斗概率模型
function calculateBattleResult(player: DerivedStats, enemy: EnemyStats, luck: number): BattleResult {
  const powerRatio = calculatePowerScore(player) / calculatePowerScore(enemy);
  const luckBonus = (luck - 50) * 0.005;  // -0.25 ~ +0.25
  const random = Math.random() * 0.3;     // 随机因子 0~0.3

  const winScore = powerRatio + luckBonus + random;

  if (winScore >= 1.6) return 'GREAT_WIN';     // 大胜
  if (winScore >= 1.2) return 'WIN';           // 小胜
  if (winScore >= 0.8) return 'DRAW';          // 平局
  if (winScore >= 0.5) return 'RETREAT';       // 败退
  return 'DEATH';                              // 重伤/战死
}

heartbeat{ ts }30s 心跳保活
kicked{ reason }被其他设备顶替

连接管理

  • Redis 存储活跃连接映射：sse:{saveId} → connectionId
  • 新连接建立时检查是否已有连接，有则发送 kicked 事件后断开旧连接
  • 心跳 30s/次，客户端断开后 60s 容忍窗口
  • 超时后冻结游戏时间线，重连后从断点继续

5.2 JSON 字段设计

设计原则：高频查询字段独立为列；低频、结构多变的数据使用 JSON 字段。
- game_state.state：最核心的 JSON 字段，存储完整结构化状态，避免频繁加列
- node.choices：选项数组，结构固定但数量可变（2-3个）
- node.state_changes：每个节点产生的状态变更指令
- world.factions / rules：世界设定，读多写少

八、容错与兜底策略

场景策略实现方式
Agent JSON 解析失败 自动重试 2 次（附带严格 JSON 提示） LangChainProvider.invokeWithRetry()
Agent 重试仍失败 降级为 Mock 模板（按境界返回保底节点） MockProvider.getFallback(realm)
Agent3 校验失败 重新生成 2 次，仍失败放行并标记"待校验" ContentValidator.validateWithRetry()
SSE 断开 60s 容忍窗口，超时冻结时间线 Redis TTL + 定时检查
多端同时连接 后开者顶替先开者 Redis sse:{saveId} 互斥锁
请求频控 写接口 1s 1次 / userId+saveId Redis sliding window
重投滥用 24h 内最多 3 次 Redis roll:{userId} incr + TTL
状态不一致 后端为唯一状态源，Agent 只提供指令不直接修改 所有 stateChanges 由后端验证后执行
LLM 超时 设置 30s 请求超时，超时返回 Mock 兜底 LangChain timeout 配置 + catch
