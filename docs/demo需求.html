零、Demo 定位

本 Demo 的目标是快速验证 AI 叙事的核心循环：角色在固定世界观下，由 AI 持续生成剧情，在关键节点提供选项供玩家抉择，选择后 AI 继续生成后续剧情，直至走向结局。

Demo 不做：境界修炼体系、战力系统、天劫、飞升、多世界、SSE 实时推送、多存档。——这些留给正式版。

前端框架
React (Vite)
后端框架
NestJS (TypeScript)
数据库
MySQL 8.0
运行环境
Docker Compose
AI 调用
大模型 API 生成叙事 + 节点
角色
固定主角（不随机）
世界观
固定凡界修仙背景
节点
AI 按剧情节奏自动触发
结局
通关 / 战死 / 寿尽
平台
Web 浏览器

一、技术栈与环境

Demo 采用前后端分离架构，Docker 统一管理运行环境，MySQL 持久化游戏状态与叙事历史。

1.1 技术选型

层级技术说明
前端React 18 + Vite单页应用，Hooks 状态管理
后端NestJS (TypeScript)模块化架构，AI 调用独立封装
数据库MySQL 8.0存储游戏会话、节点历史、叙事记录
环境Docker Compose前端 + 后端 + MySQL 三容器编排
AI 服务大模型 API（待定）叙事生成与节点生成

1.3 数据库表（极简）

表名用途核心字段
game_session 游戏会话 id, character_name, realm, state_json, status(active/ended), created_at
story_node 叙事节点 id, session_id, node_type, narrative_text, choices_json, chosen_index, created_at
Demo 仅需 2 张表。game_session 的 state_json 以 JSON 字段存储主角状态快照；story_node 记录每一步叙事与选择。

二、世界观与角色

2.1 世界观（固定）

一个典型的修仙凡界。有门派、散修、妖兽、秘境。灵气充裕但危机四伏。由后端预设一段简短的世界设定描述，作为 AI 生成叙事的基础上下文。

2.2 主角（固定）

属性值说明
姓名李青云固定，无需玩家自定义
境界练气中期初始境界
身份散修无门无派，自由修行
Demo 阶段主角完全固定，跳过随机创角环节。正式版再引入属性随机与多存档。

三、核心叙事循环

3.1 流程图

进入游戏→ 展示世界观→ AI 生成剧情→ 剧情段落展示→ 触发节点→ 展示选项→ 玩家选择→ AI 继续生成→ ……→ 结局

3.2 每一步详解

1
初始化

前端加载固定世界观文本和固定主角信息，展示给玩家。

后端维护主角的极简状态：

{
  "name": "李青云",
  "realm": "练气中期",
  "storyPhase": "intro",     // intro / early / mid / late / ending
  "history": []              // 已走过的节点摘要
}
2
AI 生成剧情段落

前端调用后端接口，后端将当前状态 + 世界观上下文 + 历史摘要发送给大模型 API，AI 返回一段 200~400 字的叙事文本。

剧情段落是"纯叙述"，不包含选项。AI 在段落末尾判断是否到达"节点触发时机"。

3
节点触发

AI 判断当前适合触发的节点类型（事件 / 战斗 / 奇遇 / 机缘），生成 2~3 个选项供玩家选择。每个选项附带简短提示。

4
玩家选择 → AI 继续

玩家选择一个选项后，AI 根据该选择生成后续叙事文本，剧情继续推进。选择产生的影响由 AI 在后续叙事中自然体现。

Demo 不做结构化状态变更（属性增减等），选择的影响完全交由 AI 在叙事层面体现。正式版再引入数值系统。
5
结局判定

AI 在生成过程中自行判断是否到达结局。三种结局类型：

结局触发条件（AI 自行判断）
飞升主角历经磨炼突破至大乘，白日飞升
陨落战斗落败、身死道消
寿尽修行至晚年仍无法突破，自然坐化

到达结局后 AI 生成一段总结性结局文本，游戏结束。

四、节点类型

Demo 保留四种核心节点，均由 AI 按当前剧情阶段自行判断触发时机。

类型说明示例
事件 日常修仙生活，财侣法地四维 坊市交易、道友切磋、偶遇散修
战斗 遭遇对手，需选择策略 妖兽袭击、仇敌追杀、擂台比试
奇遇 低概率高收益，改变命运 上古洞府、天材地宝、传承机缘
机缘 与 NPC 或剧情关联的命运锚点 师门召唤、宿敌再遇、预言应验
节点触发概率由 AI 在上下文中自行把握，Demo 不做精确频率控制。

五、API 接口

Demo 仅需 4 个接口，极简设计。

方法路径说明请求返回
POST /api/game/start 初始化游戏，返回世界观和角色 — { worldIntro, character, openingStory }
POST /api/game/next 获取下一段叙事（AI 生成） { history[] } { narrative, hasNode?, node?, isEnding?, ending? }
POST /api/game/choose 玩家选择后获取后续叙事 { nodeId, choiceIndex, history[] } { narrative, stateUpdate, hasNode?, node?, isEnding?, ending? }
GET /api/game/history 获取完整叙事历史 — { history[] }

返回字段说明

字段类型说明
narrativestringAI 生成的叙事文本（200~400字）
hasNodeboolean叙事末尾是否触发节点（若有则附带 node）
nodeobject节点信息：type, text, choices[{text, hint}]
isEndingboolean是否到达结局
endingobject结局信息：type, narrative, summary
stateUpdateobject选择后的状态变更（Demo 仅记录阶段标记）

六、前端界面

6.1 页面结构

单页面游戏，无需多路由。界面分为三个区域：

  1. 顶部状态条：主角姓名、境界、当前阶段
  2. 中部叙事区：AI 生成的修仙故事文字，自动追加展示
  3. 底部选择区：节点触发时显示 2~3 个选项按钮；非节点时显示"继续"按钮

6.2 视觉风格

  • 深色底色 + 宣纸纹理感
  • 字体：霞鹜文楷（或系统衬线体回退）
  • 叙事文字以"打字机"效果逐字展示
  • 选项按钮高亮为蓝色/紫色渐变

6.3 交互流程

进入页面 → 调用 /api/game/start → 展示开篇故事
    ↓
用户点"继续" → 调用 /api/game/next → 追加叙事文字
    ↓
(无节点) → 继续点"继续" → 循环
(有节点) → 展示选项按钮 → 用户点击选择
    ↓
调用 /api/game/choose → 追加叙事 + 新状态
    ↓
(未结局) → 循环 / 继续
(结局)   → 展示结局文本 + "重新开始"按钮

七、MVP 清单

#功能说明
1Docker 环境docker-compose.yml 编排 MySQL + 后端 + 前端三容器
2数据库初始化MySQL 建表（game_session / story_node），NestJS TypeORM 连接
3后端骨架NestJS 项目搭建，4 个接口骨架
4AI 对接对接大模型 API，封装 Prompt 模板
5叙事生成/api/game/start + /api/game/next 调用 AI 生成剧情
6节点生成AI 在叙事末尾判断是否触发节点，生成选项
7选择处理/api/game/choose 接收选择，AI 继续生成后续
8结局判定AI 自行判断结局时机，生成结局文本
9前端界面React 单页：状态条 + 叙事区 + 选择区
10打字机效果叙事文字逐字展示
11联调测试前后端联调，跑通一条完整故事线
Demo 成功标准：从"开始"到"结局"，一条完整的 AI 修仙叙事线跑通，中间至少出现 3~5 个节点选择。
